Robin von der Rijst - Wagtail in practice

You'll hear tales from the trenches.

Customers want wordpress. We ask them. You really need WP? Common misconceptions:

  • cheaper
  • quicker
  • better support.

For sure, with the right Template, is fine. But custoemrs don't come to us if they just need a template.

Why wagtail:

  • Very intuitive,
  • Content Type

Why not:

  • When client already has something.
  • When they want an off the shelf solution
  • When they need a webshop.

How we use it. Some tricks I don't get yet becaue I don't do wagtail. SinglePagePerSiteMixin, for pages that should only exist once. After that, it does not show as addable anymore.

Common pattern: Page model hierarchy:

  1. ProjectBasePage Common page attributes.
  2. Base page with our without visual.
  3. Mixins Pageintro, pagepromo...
  4. ProjectImage.

Menus:

  • Mainmenu, footermenu Two solutions
    1. Custom menu app where you manage the menus per site
      • Easy to create links to things not a page
      • More suitable for flat menus.
    2. Extend the page model
      • add menu_option to use together with built-in show_in_menus
      • Good for hierarchy.

page promotion:

  • Create a default representation for a page when shown somewhere else:
    • eg search results,
    • related pages footer,
    • facebook sharing. Offers max control for editors. Add as mixin, if customer says nothing, use default, else, use custom setting.

Wagtail the bad No file upload in forms.

  • Needs custom form. No drag'n'drop in streamfield
  • Use arrow buttons Cannot link to page preview (editor looks at preview, tries to copy link.) Hallo.js looks nice, but... (everybody laughs..., except me)
  • Swap it with another one. Streamfield with a lot of content gets a bit messy. No hierarchy for media/document
  • No folders in admin
  • No folders on disk ModelAdmin: not quite up to par yet.
  • No admin bulk actions
  • no support for drag'n'drop sortable models. easy_thumbnails, feature focal_image.

The ugly not invented here

  • wagtail thumbnails
  • django polymorphic (No idea) View logic in (Page) models
  • Undoing the mvc pattern
  • verry un-dkango-y

Matt Westcott - The Zen of Wagtail

Early days, not a lot of advance planning.

I finally have to write down the guiding principles for developing wagtail

Always wear the right hat.

  • Developer
  • Content Editor
  • Designer
  • Site admin

Let programmers program by writing programs. Screenshot of page to set fields. Programmers don't need to configure fields ttw. Shows a class diagram of what is needed for ttw field creation. Everybody is crying.

SiteAdmin doesn't define content types, programmers do.

wagtail forms example of an ui targeted at content editors. special forms, let it make the programmer. Plone can handle that.

Wagtail is just the top of the stack. Don't forget about the rest.

Content management is about tranferring information from peopls brains into the db as fast as possible.

Deliver complexity when it's needed, not before.
